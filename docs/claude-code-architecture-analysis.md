# Claude Code 源码架构分析 → terminal-agent 借鉴

> 基于 2026-03-31 泄露的 Claude Code v2.1.88 源码（~1900 文件，51万行 TypeScript）
> 综合多个社区分析报告提炼，专注于对 terminal-agent 有实际参考价值的设计模式

---

## 1. Agent Loop（核心引擎）

### Claude Code 的设计
- **单线程 master loop**，内部代号 `nO`
- 核心循环：`while(tool_call) → execute tool → feed results → repeat`
- 模型输出纯文本时循环自然终止，等待下一次用户输入
- **扁平消息历史**，没有复杂线程
- 最多允许 **1 个 sub-agent 分支**（不是多 agent 并行）
- 设计哲学：简单可调试 > 复杂并行

### 典型执行链
```
用户请求 → 搜索代码(grep) → 读文件(read) → 修改代码(edit) → 跑测试(bash) → 最终回答
```

### 对 terminal-agent 的借鉴
- [ ] **agent loop 必须有硬编码最大轮次**，防止失控循环
- [ ] **终止条件明确**：模型输出无 tool_call 时自然终止
- [ ] **扁平消息历史**优先，不要过早引入复杂线程模型
- [ ] **执行链透明**：每步操作可审计，设备操作更需要 trace

## 2. 实时干预（h2A 异步双缓冲队列）

### Claude Code 的设计
- `h2A` 是一个异步双缓冲队列
- 支持**暂停/恢复**
- 用户可以在任务执行中**注入新指令**，不需要重启
- 配合流式输出，实现真正的交互式体验

### 对 terminal-agent 的借鉴
- [ ] **设备运维场景更需要这个**：设备在执行巡检时，管理员需要能随时打断/追加指令
- [ ] 实现方式：异步消息队列 + 流式响应
- [ ] 优先级：高优先级告警可以打断当前巡检流程

## 3. 工具系统（插件化架构）

### Claude Code 的设计
- **~40 个工具**，统一 JSON 调用 → 沙箱执行 → 文本返回
- 分类：
  - **读取/发现**：Read（默认~2000行）、Glob、Grep（正则，非向量搜索）
  - **编辑**：Edit（diff 补丁）、Write（整文件）
  - **执行**：Bash（持久工作目录，但环境变量不持久）
  - **Web**：WebFetch（安全限制：只能访问用户提到的或项目内的 URL）
  - **代码智能**：LSP 集成（跳转定义、查找引用、类型检查）
- **关键决策**：用正则搜索而非向量数据库/Embeddings（简单 > 复杂）
- 工具命名空间是**治理词汇表**——权限规则、hook 匹配、subagent 白名单都引用工具名

### 对 terminal-agent 的借鉴
- [ ] **统一工具接口**：`ToolCall → Execute → ToolResult` 三段式
- [ ] **工具分类**：
  - 设备查询类（状态、日志、配置读取）
  - 设备操作类（重启、配置修改、固件升级）
  - 诊断类（网络测试、性能分析）
  - 外部集成类（监控系统、工单系统）
- [ ] **正则 > Embeddings**：设备日志搜索用正则就够了，不需要搞向量搜索
- [ ] **工具名 = 安全策略的基本单元**：权限、审计、hook 都基于工具名做匹配
- [ ] **Bash 环境变量不持久**这个设计很聪明——避免隐式状态依赖

## 4. 规划与受控并行

### Claude Code 的设计
- **TodoWrite 工具**：创建结构化 JSON 任务列表（ID、内容、状态、优先级）
- UI 渲染为可交互 checklist
- 不支持部分更新，每次更新整个列表
- 系统消息会在工具调用后注入当前 TODO 状态，防止模型跑偏
- **plan mode**：只读探索 → 写计划文件 → 用户批准 → 执行
- **Sub-agent**（dispatch_agent / I2A / Task Agent）：
  - 独立上下文窗口
  - **不能 spawn 自己的 sub-agent**（防止递归爆炸）
  - 常用于：宽搜索、多方案并行尝试
  - 结果作为普通 tool output 回到主 loop

### 对 terminal-agent 的借鉴
- [ ] **设备运维任务拆解**：复杂运维任务 → TODO list → 逐项执行
- [ ] **plan mode 对设备操作极其重要**：先出执行计划，用户确认，再执行
- [ ] **sub-agent 不能递归**：设备诊断场景下，子 agent 做专项分析，但不能再派生
- [ ] **TODO 状态注入**：防止 agent 在长任务中丢失目标

## 5. 权限模型（真正的控制平面）

### Claude Code 的设计（6 级）

| 模式 | 无需确认可做 | 适用场景 |
|------|-------------|---------|
| default | 只读 | 敏感工作、首次使用 |
| acceptEdits | 读+编辑 | 代码迭代 |
| plan | 读+规划 | 研究和设计 |
| auto | 所有+后台安全检查 | 长任务（需 classifier 审查） |
| bypassPermissions | 所有，无检查 | 隔离容器/测试 VM |
| 仅预批准工具 | 锁定策略 | 企业管控 |

- **auto mode 独特设计**：单独的 classifier 模型审查每个 action，拦截越权和注入
- 进入 auto mode 时**自动丢弃宽泛的 allow 规则**（如 `Bash(*)`）
- **规则语法**：`Bash(git diff *)`、`WebFetch(domain:example.com)`
- **deny > ask > allow** 优先级
- **4 级配置范围**：Managed（组织）→ User → Project（.claude/ 团队共享）→ Local

### 对 terminal-agent 的借鉴（核心！）
```
设备操作权限分级：

Level 0 - 只读：状态查询、日志读取、配置查看
Level 1 - 安全操作：日志清理、配置备份
Level 2 - 需确认：配置修改、服务重启
Level 3 - 危险操作：固件升级、系统重置、数据清除
Level 4 - 毁灭性：格式化、恢复出厂设置
```

- [ ] **plan mode 是设备运维的刚需**：先出方案，人工确认
- [ ] **deny > ask > allow** 优先级设计直接复用
- [ ] **每次危险操作前的二次确认**不是可选，是必须
- [ ] **dry-run 模式**：类似 plan mode，预览变更但不执行

## 6. 沙箱执行

### Claude Code 的设计
- macOS 用 **Seatbelt**，Linux/WSL2 用 **bubblewrap`
- 所有子进程继承沙箱边界
- 两种模式：
  - auto-allow：沙箱内命令自动放行，沙箱外走正常审批
  - regular：即使在沙箱内也要审批
- 可配置：允许写路径、拒绝读路径、允许域名、禁止域名
- **escape hatch**：命令被沙箱拦住后，Claude 可以用 `dangerouslyDisableSandbox` 重试（仍需审批）

### 对 terminal-agent 的借鉴
- [ ] **设备操作沙箱**（dry-run 模式）：
  - 预览即将执行的命令
  - 模拟配置变更的影响
  - 不真正写入设备
- [ ] **操作范围限定**：只能操作被授权的设备/命令
- [ ] **子进程继承边界**：agent 调用的脚本/工具也要受约束

## 7. 记忆系统（深度）

### 7.1 CLAUDE.md —— 写给 AI 的"入职手册"

#### 四层加载机制
| 层级 | 来源 | 范围 | 优先级 |
|------|------|------|--------|
| ① | 公司统一下发 | 所有人共用（代码规范） | 最低 |
| ② | 个人全局配置 | 所有项目生效（注释风格） | ↑ |
| ③ | 代码仓库里 | 团队共享，跟项目走 | ↑ |
| ④ | 本地私有 | 不提交仓库，只对自己 | 最高 |

- **优先级反转**：距离越近的越优先（④ > ③ > ② > ①）
- 启动时自动从当前目录向上查找（monorepo 友好）
- 子目录的 path-scoped 规则在读取该目录文件时自动加载
- 不是个人笔记，是**仓库级别的策略输入**

### 7.2 记忆的 4 种类型

| 类型 | 内容 | 示例 |
|------|------|------|
| **用户记忆** | 你是谁、擅长什么、沟通偏好 | 资深工程师 vs 小白，协作方式不同 |
| **反馈记忆** | 纠正过什么、认可过什么 | "别写总结"、"对，就这样" |
| **项目记忆** | 当前任务、负责人、deadline | 在做什么、进展如何 |
| **参考记忆** | 哪里能找到什么资料 | Jira、监控大盘、文档链接 |

#### 关键原则：记错误，也记认可
- 源码原文：**"Record from failure AND success. If only save corrections, you will avoid past mistakes but drift away from approaches the user has already validated, and may grow overly cautious."**
- 只记纠正 → AI 越来越保守，远离已验证有效的方法
- 不说话但没推翻 = 默认认可信号
- **"Corrections are easy to notice; confirmations are quieter — watch for them."**

#### 记忆格式：规则 + Why + How to apply
- 不只记"是什么"，还记**"为什么"**
- **"Knowing why lets you judge edge cases instead of blindly following the rule."**
- 知道原因才能灵活处理边界，不知道原因只能机械执行

#### 时间处理
- 所有时间存**绝对时间**，不存"下周四"而是"2026年4月10日"
- 三个月后相对时间不仅失效，还会添乱

### 7.3 上下文压缩（三重机制）

| 级别 | 触发条件 | 策略 |
|------|---------|------|
| 微压缩 | 空间还够 | 轻度整理，去重复 |
| 自动压缩 | 中度紧张 | 提炼关键信息，压缩对话轮次 |
| 完整压缩 | 空间告急 | 全面重写，保留骨架 |

- 触发线 = 上下文窗口总量 - 13000 Token 缓冲
- **必须保留**：改了哪些文件、踩过的错误、用户所有反馈
- 任务未结束时，**必须原文引用用户最近的话**（一字不差）
  - "把这个功能改成异步的" vs "把这个功能优化一下" → 执行方向完全不同
- 压缩后 CLAUDE.md 内容自动重新加载（skill 列表是例外）
- `compact` 命令手动触发，`context` 命令查看空间占用
- **Compact Instructions** 可写在 CLAUDE.md 里控制压缩保留什么

#### 工程教训
- 每天 1279 个 session 连续压缩失败超过 50 次，最严重的失败 3272 次
- 浪费约 25 万次 API 调用
- 修复方案：3 行代码，连续失败 3 次就停止重试

### 7.4 底层存储：文件系统，不是向量数据库

- 每条记忆是**独立 Markdown 文件**
- MEMORY.md 是索引，最多 200 行，每条 ≤150 字
- 人可以直接打开看、手动改、git 管理、diff、回滚、grep
- **"可以被人类审查的记忆，信任成本低得多"**
- 用向量数据库你不知道 AI 记住了什么，用 Markdown 打开就看到

### 7.5 AutoDream（睡眠整理）

- 触发条件：距上次整理 **≥24 小时**，且期间 **≥5 个工作 session**
- 提示词第一行：**"You are performing a dream — a reflective pass over your memory files."**
- 4 个阶段：
  1. **定向**：先搞清楚自己现在知道什么
  2. **收集**：扫描今天的日志，找值得留下来的新信息
  3. **整合**：新信息写进记忆，相对时间 → 绝对时间，新发现推翻旧记忆就直接修正
  4. **修剪**：更新索引，删过时的，保持 200 行以内
- **记忆漂移**（memories that drifted）：当时对的，现在情况变了 → 找到并修正
- 不是简单叠加新内容，是**维护记忆的准确性**
- 人脑睡眠时也是这样——整理白天信息，去掉过时的，保留重要的

### 7.6 Checkpoints（检查点）
- 每次编辑前快照文件状态
- 可恢复：代码、对话、或两者
- 跨会话持久化
- ⚠️ **Bash 修改的文件不被 checkpoint 追踪**（`echo "xxx" > file` 绕过了编辑工具）

### 7.7 Sub-agent 的记忆
- Sub-agent 有**独立的 auto memory**
- 文件读取在自己的上下文窗口内，只返回摘要
- 不会把大量中间内容拖回主会话

### 对 terminal-agent 的借鉴

#### 记忆架构设计
```
四层记忆（参考 CLAUDE.md 加载机制）：

1. 组织级：公司运维规范、安全策略（所有人共用）
2. 项目级：当前项目的架构、特殊配置（团队共享）
3. 设备级：每台设备的"病历本"（历史故障、特殊处理）
4. 个人级：管理员的操作习惯、偏好（本地私有）

优先级：设备级 > 项目级 > 组织级（越具体越优先）
存储格式：Markdown 文件，不用数据库
```

#### 记忆类型映射
| Claude Code | terminal-agent | 内容 |
|-------------|---------------|------|
| 用户记忆 | 管理员偏好 | 谁习惯什么方式沟通、操作风格 |
| 反馈记忆 | 纠正+认可 | "这种重启方式不对"、"对，就这样处理" |
| 项目记忆 | 运维任务 | 当前在做什么、谁负责、deadline |
| 参考记忆 | 资料索引 | 监控在哪、工单在哪、文档在哪 |

#### 压缩策略
- 长运维会话 → 三重压缩（微/自动/完整）
- 必须保留：改了哪些配置、踩过的坑、管理员的反馈
- 任务未结束时，原文保留管理员最近的指令（一字不差）

#### AutoDream 适配
- 每天自动整理：扫描运维日志 → 提取有价值信息 → 更新设备病历
- 记忆漂移检测：设备换了固件、改了配置 → 修正旧记忆
- 索引维护：删除过时告警记录，保持精简

#### 具体实现建议
- [ ] **设备病历本**：`devices/{device_id}/MEMORY.md`（设备级记忆索引）
- [ ] **运维日志**：`memory/YYYY-MM-DD.md`（每日运维记录）
- [ ] **自动记忆提取**：后台 agent 分析对话，提取值得记住的运维经验
- [ ] **记好也记坏**：不只记故障，也记成功的处理方案
- [ ] **时间用绝对时间**：不存"上次"，存"2026-04-03 14:30"
- [ ] **压缩保留原话**：管理员的关键指令压缩时原文保留
- [ ] **每晚 AutoDream**：自动整理当天运维经验，更新设备病历
- [ ] **Checkpoint**：配置修改前自动备份
- [ ] **⚠️ 命令行修改额外备份**：Bash 操作不受 checkpoint 保护，需单独处理

## 8. 多 Agent 协作（含 Coordinator 模式）

### 8.1 Coordinator 模式（管理架构）
- 一个 AI 当 **Coordinator（项目经理）**，管一群 AI worker
- Worker 负责：研究、写代码、测试、验证
- Coordinator 负责：**拆任务、派活、验收**
- 核心逻辑全在 **370 行自然语言 prompt** 里，不在代码里

#### 关键管理原则（来自源码 prompt）
- **"永远不要把理解力外包给下属"**
  - 不要写"根据你的发现去修复这个 bug"
  - 必须自己理解研究结果，然后写清楚：改哪个文件、第几行、为什么改
- **验收不是橡皮图章**
  - "验收意味着证明代码能用，不是确认代码存在"
  - 要跑测试、查类型、试边界情况
  - 如果什么看起来不对，要深挖，不要轻描淡写"应该没关系"

### 8.2 Sub-agent 隔离机制
- Sub-agent 有**独立上下文窗口**
- 支持自定义 prompt、工具白名单/黑名单、模型选择、权限模式
- **worktree 隔离模式**：sub-agent 在临时 Git worktree 中工作（爆炸半径控制）
- 插件 sub-agent 不支持某些 frontmatter 字段（安全边界）
- Agent SDK 暴露相同的工具循环和上下文管理

### 对 terminal-agent 的借鉴

#### Coordinator 模式映射
```
主 Agent（Coordinator）          Sub-agent（Worker）
├─ 理解运维任务                    ├─ 设备诊断 agent
├─ 拆解子任务                      ├─ 配置审查 agent
├─ 分派给 sub-agent               ├─ 日志分析 agent
├─ 验收结果（不是橡皮图章）         └─ 修复执行 agent
└─ 汇报给管理员
```

- [ ] **Coordinator 必须自己理解诊断结果**，不能直接转发给管理员
- [ ] **验收要验证**：跑测试、检查配置、确认设备状态，不是"看起来没问题"
- [ ] **每个 sub-agent 只拿需要的工具**（工具白名单原则）
- [ ] **诊断 agent**：只读权限 + 网络诊断工具
- [ ] **修复 agent**：写权限，但在 dry-run 环境先验证
- [ ] **审查 agent**：只读 + diff 工具，审查配置变更建议

## 9. Hooks（策略引擎）

### Claude Code 的设计
- 用户定义的 shell 命令/HTTP 端点/LLM prompt/agent hook
- 在 Claude Code 生命周期的特定点执行
- **PreToolUse**：工具调用前触发，可以 allow/deny/ask/defer/修改输入/追加上下文
- **PermissionRequest**：权限弹窗前触发，可以代替用户审批
- 多个 hook 意见不一致时有优先级规则

### 对 terminal-agent 的借鉴
- [ ] **PreToolUse hook = 设备操作的最后防线**
  - 检查命令是否在白名单内
  - 检查设备当前状态是否允许操作
  - 检查操作时间是否在维护窗口内
- [ ] **团队可以编码运维策略为 hook**，而不是靠文档
- [ ] **hook 匹配基于工具名**——再次印证工具命名空间的重要性

## 11. 缓存经济学（Token 就是钱）

### Claude Code 的设计
- 追踪了 **14 种缓存失效原因**
- 有一个函数叫 `DANGEROUS_未缓存系统提示区`——碰之前想清楚，改了缓存就炸
- 优化案例：把 Agent 列表从工具描述移到对话消息 → 省了 **10.2% 的缓存创建 Token**
- 每天 25 万次无效 API 调用，3 行代码修复（连续失败 3 次停止重试）

### 对 terminal-agent 的借鉴
- [ ] **缓存策略 = 财务问题**：每轮 API 调用都要考虑缓存命中率
- [ ] **设备信息不要放在 system prompt 里**：放对话消息，利用 prompt caching
- [ ] **失败重试有上限**：连续失败 N 次停止，别烧钱
- [ ] **追踪缓存失效原因**：至少记录 top 5 的失效场景

## 12. YOLO 分类器（AI 驱动的风控）

### Claude Code 的设计
- 1500 行代码专门做一件事：**用 AI 判断操作该不该自动放行**
- 不是黑白名单，而是把对话上下文 + 用户配置 + 操作内容全部扔给模型
- 输出：低风险自动放行 / 高风险弹确认 / 极高风险直接拦截

### 对 terminal-agent 的借鉴
- [ ] **设备操作可以用小模型做风险评估**
  - 输入：操作类型 + 设备状态 + 历史记录 + 当前时间
  - 输出：风险等级（自动放行 / 需确认 / 拦截）
- [ ] **不是替代权限系统，是补充**：权限系统管"能做什么"，YOLO 管"现在做合不合适"

## 13. KAIROS（后台 Agent / AI 操作系统雏形）

### Claude Code 的设计
- AI 可以 **7×24 后台运行**
- 自设定时任务，监听 GitHub 事件（PR、CI 完成）
- 主动发推送通知
- 每天自动做**"记忆蒸馏"**——把一天学到的整理归档

### 对 terminal-agent 的借鉴

- [ ] **事件驱动**：监听设备事件（重启、告警、配置变更），自动响应
- [ ] **每日记忆蒸馏**：自动整理当天运维经验，更新设备病历

## 14. 反蒸馏 & 安全细节

### Claude Code 的设计
- **反蒸馏**：偷录 API 流量 → 注入假的工具定义污染训练数据
- **Undercover Mode**：AI 写开源代码时自动隐藏 AI 痕迹，commit message 写得跟人一样
- 注释："There is NO force-OFF"——没有强制关闭开关

### 对 terminal-agent 的借鉴
- 这块更多是商业/安全考量，不直接影响架构
- 但可以关注：设备操作日志中是否需要隐藏 AI 参与痕迹（合规场景）

## 10. 关键设计哲学

### "Do the simple thing first"
- 正则 > Embeddings
- Markdown 文件 > 数据库
- 单线程循环 > 多 agent 并行
- 扁平历史 > 复杂线程

### "Architectural behaviors > prompt behaviors"
- 工具可以被 deny
- Sub-agent 有自己的上下文窗口
- Hook 可以拒绝 action
- 沙箱可以在 OS 级别阻断子进程
- **这些是系统属性，不是 prompt 装饰**

### "Bounded failure"
- 不假设模型永远正确
- 沙箱 + 权限 + hook 三层防护
- 即使 prompt injection 操控了行为，沙箱仍能防止关键破坏
